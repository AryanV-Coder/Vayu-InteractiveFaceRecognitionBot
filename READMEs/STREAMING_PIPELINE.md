# Real-Time Voice Pipeline Architecture

This document explores the structural decisions behind the Vayu Interactive Face Recognition Bot pipeline, specifically examining when to use data streaming versus blocking (batch) execution across the STT, LLM, and TTS components.

## Pipeline Overview
The core audio loop of the bot operates in three steps:
1. **STT (Speech-to-Text):** Converting user audio into text.
2. **LLM (Language Model):** Generating a response based on the transcribed text.
3. **TTS (Text-to-Speech):** Converting the generated text back into playable audio.

---

## 1. Speech-to-Text (STT)

### How it currently works (`Blocking/Batch`)
The pipeline runs a background thread that constantly listens to the microphone. 
- It uses **Silero VAD** (Voice Activity Detection) to identify when speech starts.
- It buffers the audio until ~1.0 second of silence is detected.
- It saves this buffer as a `.wav` file and sends the **entire file** to the STT model (Whisper/Sarvam) at once.

### The Streaming Alternative
Streaming STT means sending audio chunks via WebSocket to an STT API *as the user speaks*. The API continuously returns partial transcriptions (e.g., "Hello", "Hello how", "Hello how are you?").

### When to use Streaming STT:
✅ **To interrupt users mid-sentence.** If the bot needs to detect specific keywords (e.g., "Stop") instantly to interrupt the user.
✅ **Displaying Live Captions.** If your UI needs to show the user's words appearing on-screen in real-time as they speak.

### Why we do NOT use it in Vayu:
🚫 **Context Window Dependency.** Even if we get the partial transcription ("Hello how are..."), we **cannot** send it to the LLM yet. The LLM needs the full, completed sentence to understand the context and generate a proper, coherent response.
🚫 Therefore, streaming STT adds massive architectural complexity (WebSockets, continuous connection management) but yields **zero latency benefit** because we must wait for the user to finish speaking anyway before the LLM can start thinking.

**Decision:** Stick to VAD + Batch STT. It is simpler, more stable, and functionally equivalent in end-to-end latency for our use case.

---

## 2. Large Language Model (LLM) Generation

### How it currently works (`Blocking/Batch`)
The STT outputs a single string (e.g., "Tell me about the fest."). This string is sent to Groq. The system waits until Groq completes the **entire paragraph** of the response before moving to the TTS step.

### The Streaming Alternative
Streaming LLM means receiving the output token-by-token. 
For example, the LLM starts returning: `"Welcome " -> "to " -> "Converge!"` immediately, while it is still computing the next sentence.

### When to use Streaming LLM:
✅ **Long Responses.** If the LLM generates a 300-word monologue, waiting for the massive string to finish generating will cause a huge, unnatural pause.
✅ **Pipelined TTS.** When combined with Streaming TTS (see below), you can send the first 5 words generated by the LLM directly to the TTS engine so the bot starts talking almost instantly.

### Why we do NOT currently use it in Vayu:
🚫 **Architectural Constraint.** By strict system prompt design, Vayu's LLM responses are capped at **1 or 2 short sentences max**. 
Because Groq processes tokens at ~800 tokens per second, generating 2 short sentences takes literally **milliseconds** (< 0.5s). The overhead and complexity of chunking those tokens grammatically to feed to the TTS engine outweighs the fraction-of-a-second latency gained.

**Decision:** Since Groq is utilized for extremely short responses, Batch LLM is preferred. If the bot's prompt is ever expanded to give long, multi-paragraph speeches, we MUST switch to Streaming LLM + Chunked TTS.

---

## 3. Text-to-Speech (TTS)

### How it *used* to work (`Blocking/Batch`)
The script sent the full LLM text to Sarvam, waited for Sarvam to generate the complete MP3 file, saved it to the hard drive, loaded it back into memory, and then played it. This caused a massive, unnatural delay.

### The Streaming Alternative (`Currently implemented in Vayu`)
The script establishes a WebSocket connection with Sarvam AI. 
1. We send the *full* LLM text block to Sarvam via the WebSocket all at once.
2. Sarvam immediately begins translating the start of the text into audio and pipes those audio chunks back to us in base64.
3. Our code instantly decodes the chunk and pipes it directly into the `ffplay` audio player's memory buffer. 

### Why we DO use it in Vayu:
✅ TTS generation is computationally heavy and slower than LLM text generation. 
✅ By streaming the audio chunks directly into the player buffer, the "Time to First Byte" mapping is drastically reduced to **~0.4 seconds**. The bot starts physically speaking its first sentence out loud while Sarvam's servers are still actively generating the audio for the *end* of the sentence.

**Decision:** We use a Hybrid model. We send a **Batch Text Block** to the TTS engine, but we **Stream the Audio Response** into the media player. This provides the best synthesis quality (TTS engines perform better when they can see the whole sentence for pacing/intonation) while giving us ultra-low playback latency.
